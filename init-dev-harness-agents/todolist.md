# Harness 工程后期规划

> 本文档描述 init-dev-harness-agents 的后期演进规划，包括具体任务、时间节点和预期目标。
> 规划按优先级分为 P0（核心）、P1（重要）、P2（增强）三个等级。

---

## 一、协同机制深化（P0 — 核心）

### 1.1 进度看板可视化工具
- **目标**：将 `agents/sync/progress.yaml` 渲染为可视化看板（Web 页面或 CLI TUI）
- **时间节点**：Q3 Week 1-2
- **具体任务**：
  - [ ] 设计看板 UI 布局（工作流状态、角色队列、阻塞项、里程碑）
  - [ ] 实现 YAML → HTML 渲染器，支持实时刷新
  - [ ] 支持阻塞项高亮告警（红色标记 + 通知）
  - [ ] 支持里程碑进度条展示
- **验收标准**：打开看板即可一目了然所有角色工作状态

### 1.2 交接记录自动化
- **目标**：角色间任务交接时自动生成 handoff 记录，减少手动填写
- **时间节点**：Q3 Week 3-4
- **具体任务**：
  - [ ] 在工作流节点切换时自动触发 handoff 模板填充
  - [ ] 交接内容自动从上下文提取（summary、decisions、artifacts）
  - [ ] 接收方确认机制：未确认的交接阻塞后续节点
  - [ ] 交接超时告警：超过 N 分钟未确认自动通知 orchestrator
- **验收标准**：交接零手动，接收方必须确认才能继续

### 1.3 同步日志结构化
- **目标**：`sync-log.md` 从自由文本升级为结构化日志，支持查询与统计
- **时间节点**：Q3 Week 5-6
- **具体任务**：
  - [ ] 定义日志 JSON Schema（时间、角色、操作类型、工作流ID、详情）
  - [ ] 实现日志写入 SDK（各角色统一调用）
  - [ ] 支持按角色/工作流/时间范围查询
  - [ ] 生成周报摘要（本周各角色操作统计）
- **验收标准**：可查询任意时间段内任意角色的操作历史

## 二、经验知识库建设（P0 — 核心）

### 2.1 经验总结模板化与索引
- **目标**：`experience-summary.md` 按分类索引，支持快速检索
- **时间节点**：Q3 Week 7-8
- **具体任务**：
  - [ ] 定义经验条目标准模板（背景→决策→结果→反思→可迁移洞察）
  - [ ] 按分类建立索引：架构决策 / 协作流程 / 技术选型 / 调试技巧 / 性能优化
  - [ ] 支持标签系统（如 `#api-design` `#performance` `#collaboration`）
  - [ ] 实现跨条目关联引用（"参见 EXP-042"）
- **验收标准**：新成员可通过索引快速找到相关经验

### 2.2 FAQ 与最佳实践联动
- **目标**：FAQ（错误）与 best-practices（正向）建立双向关联
- **时间节点**：Q3 Week 9-10
- **具体任务**：
  - [ ] 每个 FAQ 条目关联对应的 best-practices 条目（"如何避免此错误"）
  - [ ] 每个 best-practices 条目关联对应的 FAQ 条目（"不这样做会出什么错"）
  - [ ] 同类 FAQ 出现 ≥3 次自动触发 best-practices 条目生成
  - [ ] 实现 FAQ → best-practices 的自动跳转
- **验收标准**：FAQ 和 best-practices 形成闭环，互相引用

### 2.3 跨项目经验迁移
- **目标**：将本项目沉淀的经验抽象为可跨项目复用的知识
- **时间节点**：Q4 Week 1-4
- **具体任务**：
  - [ ] 识别项目特定的经验 vs 通用可迁移的经验
  - [ ] 通用经验提取为独立知识卡片
  - [ ] 建立跨项目经验库（独立于单个 Harness 项目）
  - [ ] 新项目初始化时自动注入相关通用经验
- **验收标准**：新项目启动时自动获得历史项目经验

## 三、工作流引擎增强（P1 — 重要）

### 3.1 条件分支与动态路由
- **目标**：工作流支持条件判断和动态路径选择
- **时间节点**：Q4 Week 5-8
- **具体任务**：
  - [ ] 工作流 DSL 支持 `condition` 字段（如 `review.failed → fix`）
  - [ ] 支持 `parallel` 并行节点（多 executor 同时工作）
  - [ ] 支持 `debate` 辩论节点（多角色讨论达成共识）
  - [ ] 支持 `loop` 循环节点（直到条件满足）
  - [ ] orchestrator 根据运行时状态动态选择下一节点
- **验收标准**：工作流不再是线性序列，支持复杂分支逻辑

### 3.2 工作流模板市场
- **目标**：积累可复用的工作流模板，新项目快速选用
- **时间节点**：Q4 Week 9-12
- **具体任务**：
  - [ ] 从 `patterns.md` 中提取稳定模式为工作流模板
  - [ ] 模板分类：CRUD 开发 / API 设计 / 重构 / 性能优化 / 安全审计
  - [ ] 模板参数化（项目名、技术栈等变量替换）
  - [ ] 模板评分与使用统计
- **验收标准**：新项目初始化时可从模板市场选择工作流

### 3.3 工作流执行回放
- **目标**：记录工作流完整执行过程，支持事后回放分析
- **时间节点**：Q1 Week 1-4（次年）
- **具体任务**：
  - [ ] 记录每个工作节点的输入/输出/耗时/决策
  - [ ] 实现时间线回放视图
  - [ ] 支持对比分析（同类工作流不同执行的差异）
  - [ ] 识别瓶颈节点（耗时最长、失败率最高）
- **验收标准**：可回放任意历史工作流执行过程

## 四、角色能力升级（P1 — 重要）

### 4.1 角色能力矩阵
- **目标**：建立角色能力评估体系，量化各角色成熟度
- **时间节点**：Q1 Week 5-8（次年）
- **具体任务**：
  - [ ] 定义能力维度（准确率、覆盖度、响应速度、自主决策能力）
  - [ ] 每个角色定期自评 + orchestrator 评估
  - [ ] 生成能力雷达图
  - [ ] 能力短板自动触发角色定义升级
- **验收标准**：每个角色有量化的能力画像

### 4.2 角色间协作优化
- **目标**：减少角色间等待时间，提升整体吞吐量
- **时间节点**：Q1 Week 9-12（次年）
- **具体任务**：
  - [ ] 分析 `sync-log.md` 中的角色间等待时间
  - [ ] 识别可并行化的串行节点
  - [ ] 实现预判式路由（orchestrator 提前分配任务到空闲角色）
  - [ ] 实现流水线模式（角色 A 部分产出即可触发角色 B 开始）
- **验收标准**：角色间平均等待时间降低 50%

### 4.3 新角色接入框架
- **目标**：支持项目按需扩展新角色（如 tester、documenter、pm）
- **时间节点**：Q2 Week 1-4（次年）
- **具体任务**：
  - [ ] 定义角色接入规范（必须实现的接口/文件）
  - [ ] 角色注册机制（harness.yaml 中声明）
  - [ ] 新角色自动继承协同基础设施（progress、sync-log、handoff）
  - [ ] 新角色 FAQ 和 best-practices 自动初始化
- **验收标准**：新增角色只需 3 个文件即可接入 Harness

## 五、质量保障体系（P2 — 增强）

### 5.1 代码审查覆盖率追踪
- **目标**：量化 reviewer 的审查覆盖度，避免审查盲区
- **时间节点**：Q2 Week 5-8（次年）
- **具体任务**：
  - [ ] 定义审查维度（安全、性能、可读性、架构一致性、测试覆盖）
  - [ ] 每次审查记录覆盖的维度
  - [ ] 生成审查覆盖度报告
  - [ ] 覆盖度不足自动告警
- **验收标准**：每次审查至少覆盖 4/5 维度

### 5.2 回归测试自动化
- **目标**：executor 产出代码后自动触发回归测试
- **时间节点**：Q2 Week 9-12（次年）
- **具体任务**：
  - [ ] 集成测试框架（根据项目技术栈自动选择）
  - [ ] 工作流中增加 `test` 节点（executor → test → reviewer）
  - [ ] 测试失败自动回退到 executor 修复
  - [ ] 测试覆盖率门槛（低于阈值阻塞合并）
- **验收标准**：代码合并前必须通过自动化测试

### 5.3 安全扫描集成
- **目标**：reviewer 审查时自动触发安全扫描
- **时间节点**：Q3 Week 1-4（次年）
- **具体任务**：
  - [ ] 集成 `TRAE-security-review` 到标准工作流
  - [ ] 安全漏洞分级（Critical / High / Medium / Low）
  - [ ] Critical/High 漏洞阻塞合并
  - [ ] 安全扫描结果写入审查报告
- **验收标准**：每次代码审查自动包含安全扫描结果

## 规划原则

1. **P0 优先**：协同机制和经验知识库是 Harness 作为"工程系统"的基础，必须优先完成
2. **渐进增强**：每个阶段在前一阶段基础上叠加，不推翻重来
3. **数据驱动**：所有改进基于 `sync-log.md` 和 `progress.yaml` 的实际数据
4. **可回滚**：每个改进独立可开关，出问题可单独回滚
5. **大周期回顾对齐**：每完成一个 P0/P1 模块，触发一次大周期沉淀（`/harness-evolve`）
